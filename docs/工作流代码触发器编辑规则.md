# 工作流代码触发器编辑规则

代码触发器用于决定工作流是否执行。后台调度器会按照“检查间隔”唤醒工作流，并执行触发器中的 JavaScript 代码：

- 返回 `true`：继续执行与触发器连通的 AI、工具和通知模块。
- 返回 `false`：本轮不执行工作流，等待下一次检查。
- 抛出异常或代码存在语法错误：本轮执行失败，并显示“触发器代码错误”。

## 基本格式

触发器代码是一个 JavaScript 函数体，必须使用 `return` 返回判断结果：

```js
return ctx.hour === 8;
```

可以声明变量、使用条件语句或组合多个条件：

```js
const isMonday = ctx.weekday === 1;
const isEightOClock = ctx.hour === 8 && ctx.minute === 0;

return isMonday && isEightOClock;
```

代码是同步执行的，不支持 `await`、网络请求或直接调用工作流工具。需要使用工具时，请把工具模块连接在触发器之后，由后续 AI 任务调用。

## `ctx` 上下文

触发器可以读取以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `ctx.now` | `number` | 当前 Unix 毫秒时间戳 |
| `ctx.year` | `number` | 当前年份，例如 `2026` |
| `ctx.month` | `number` | 当前月份，范围 `1–12` |
| `ctx.day` | `number` | 当前日期，范围 `1–31` |
| `ctx.weekday` | `number` | 星期，周日为 `0`，周一为 `1`，周六为 `6` |
| `ctx.hour` | `number` | 当前小时，范围 `0–23` |
| `ctx.minute` | `number` | 当前分钟，范围 `0–59` |
| `ctx.second` | `number` | 当前秒数，范围 `0–59` |
| `ctx.lastRunAt` | `number \| null` | 工作流上一次真正执行的时间戳；从未执行时为 `null` |
| `ctx.minutesSinceLastRun` | `number \| null` | 距离上一次真正执行经过的整数分钟数 |
| `ctx.foregroundPackage` | `string \| null` | 最近检测到的 Android 前台 App 包名 |
| `ctx.previousForegroundPackage` | `string \| null` | 上一次检查时记录的前台 App 包名 |
| `ctx.foregroundChanged` | `boolean` | 当前包名与上一次检查记录是否不同 |

时间字段使用手机当前系统时区。

## 时间触发示例

### 每天 08:00

```js
return ctx.hour === 8 && ctx.minute === 0;
```

建议把检查间隔设置为 `1` 分钟。若检查间隔大于目标时间窗口，可能错过该次触发。

### 每周一 08:00

```js
return ctx.weekday === 1 &&
  ctx.hour === 8 &&
  ctx.minute === 0;
```

### 每周一至周五 18:30

```js
const isWeekday = ctx.weekday >= 1 && ctx.weekday <= 5;

return isWeekday &&
  ctx.hour === 18 &&
  ctx.minute === 30;
```

### 每月 1 日 09:00

```js
return ctx.day === 1 &&
  ctx.hour === 9 &&
  ctx.minute === 0;
```

### 至少间隔 6 小时

```js
return ctx.lastRunAt === null ||
  ctx.minutesSinceLastRun >= 6 * 60;
```

### 每天 08:00，且避免同一分钟重复执行

通常一分钟检查一次不会在同一分钟重复唤醒，但仍可以增加上次执行时间判断：

```js
const isTargetTime = ctx.hour === 8 && ctx.minute === 0;
const hasNotRunRecently =
  ctx.minutesSinceLastRun === null ||
  ctx.minutesSinceLastRun >= 2;

return isTargetTime && hasNotRunRecently;
```

## App 打开触发示例

读取前台 App 需要在 Android 系统设置中授予 YSClaude“应用使用情况访问权限”。未授权时，`ctx.foregroundPackage` 通常为 `null`。

### 每次切换到微信时触发

```js
return ctx.foregroundChanged &&
  ctx.foregroundPackage === 'com.tencent.mm';
```

### 每次切换到 QQ 时触发

```js
return ctx.foregroundChanged &&
  ctx.foregroundPackage === 'com.tencent.mobileqq';
```

### 打开指定 App，并且距离上次提醒超过 30 分钟

```js
const openedTargetApp =
  ctx.foregroundChanged &&
  ctx.foregroundPackage === 'com.example.app';

const cooldownFinished =
  ctx.minutesSinceLastRun === null ||
  ctx.minutesSinceLastRun >= 30;

return openedTargetApp && cooldownFinished;
```

前台 App 是周期检测，不是系统级即时回调。如果用户在两次检查之间快速打开并关闭 App，工作流可能无法捕获。需要尽快响应时，将检查间隔设为 `1` 分钟。

## 组合条件示例

### 工作日晚上打开短视频 App 时提醒

```js
const isWeekday = ctx.weekday >= 1 && ctx.weekday <= 5;
const isEvening = ctx.hour >= 20;
const openedTargetApp =
  ctx.foregroundChanged &&
  ctx.foregroundPackage === 'com.example.video';

return isWeekday && isEvening && openedTargetApp;
```

### 只在某个日期范围内执行

```js
const start = new Date('2026-08-01T00:00:00').getTime();
const end = new Date('2026-09-01T00:00:00').getTime();

return ctx.now >= start && ctx.now < end;
```

## 检查间隔

检查间隔表示后台多久运行一次触发器判断，最小为 `1` 分钟。

- 精确到分钟的提醒：建议 `1` 分钟。
- 只需大致周期检查：可设置 `5–30` 分钟以减少耗电。
- Android 省电、待机和厂商后台限制可能造成一定延迟。
- 检查间隔不是工作流执行间隔；只有代码返回 `true` 才会执行后续模块。

## 一键测试

工作流页面的“测试”按钮会跳过触发器条件，直接执行后续模块。这样可以独立验证 AI 提示、工具调用和通知是否正常。

“测试”不能验证触发器代码是否会在某个未来时间返回 `true`。编辑代码后，应先检查语法，再结合较短检查间隔进行实际测试。

## 画布连接规则

- 触发器应位于流程顶部。
- AI、工具和通知模块需要通过连线从触发器可达。
- 未与触发器连通的模块不会加入本次工作流执行。
- 当前执行器使用画布中的第一个触发器作为触发条件，建议一个工作流只放置一个触发器。

## 安全与稳定性

触发器代码由用户在本机编写并执行，具有较高自由度。请遵守以下规则：

- 不要编写死循环，例如 `while (true) {}`。
- 不要进行超大规模循环或长时间计算，否则会阻塞后台任务。
- 不要依赖未列出的全局变量或运行环境内部对象。
- 不要在代码中保存 API Key、密码等敏感信息。
- 对可能频繁满足的条件增加 `lastRunAt` 或 `minutesSinceLastRun` 冷却判断。

推荐让触发器只负责返回布尔值，把实际操作交给后续 AI 和工具模块。
